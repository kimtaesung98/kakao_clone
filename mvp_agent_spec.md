# MVP 앱 개발 에이전트 활용 명세서

**버전:** 1.0  
**작성일:** 2026-05-14  
**대상:** Claude 에이전트(Pro / Max 요금제)를 활용한 앱 MVP 개발자

---

## 1. 문서 목적

본 명세서는 Claude 에이전트를 이용하여 MVP 앱을 개발할 때 발생하는 **컨텍스트(토큰) 폭증 문제**를 방지하고, 공학적으로 체계화된 세션 운용 전략을 제공하기 위해 작성되었습니다.

> **핵심 전제:** 에이전트는 공학적 판단을 대신하는 도구가 아닙니다. 명세가 확정된 범위 안에서 구현을 수행하는 실행 도구입니다. 설계·우선순위·세션 분리는 개발자가 담당합니다.

## 3. 토큰 운용 기본 개념

### 3.1 토큰이란

- AI가 텍스트를 처리하는 기본 단위
- 영어 약 4글자 = 1토큰 / 한국어는 글자당 더 많은 토큰 소모
- **메시지 길이 + 첨부 파일 + 대화 전체 누적 기록** 모두 토큰에 포함됨

### 3.2 요금제별 주간 사용 한도 (Claude Code 기준)

| 요금제 | 월 비용 | Sonnet 4 예상 | Opus 4 예상 |
|--------|---------|--------------|------------|
| Pro | $20 | 주당 40~80시간 | — |
| Max | $100 | 주당 140~280시간 | 주당 15~35시간 |
| Max | $200 | 주당 240~480시간 | 주당 24~40시간 |

> 위 수치는 토큰 기반 측정이며, 코드베이스 크기와 작업 복잡도에 따라 실제 사용량이 달라집니다.

### 3.3 토큰 임박(Rate Limit)의 의미

- 일정 시간(5시간) 또는 주간 누적 토큰이 한도에 도달한 상태
- `"Claude usage limit reached. Your limit will reset at 12am"` 메시지로 표시됨
- **Pro:** 5시간마다 자동 리셋
- **Max:** 한도 초과 후 Extra Usage(추가 사용)를 표준 API 요금으로 구매 가능
  - Opus: 백만 토큰당 $15 / Sonnet: 백만 토큰당 $3

## 4. 컨텍스트 폭증의 원인

에이전트에게 전체 개발을 맡겼을 때 토큰이 급격히 증가하는 3가지 구조적 원인:

| 원인 | 설명 |
|------|------|
| **반복 수정 누적** | 에이전트가 실패 후 재시도를 동일 대화 안에서 반복하면, 실패 기록이 모두 컨텍스트로 쌓임 |
| **코드 전체 재로드** | 매 메시지마다 지금까지의 대화 기록 전체를 다시 읽어야 하므로, 대화가 길수록 지수적 증가 |
| **범위 미확정 작업** | 목표가 불명확하면 에이전트가 방향 탐색에 토큰을 소모하며, 수정 횟수가 증가함 |

## 5. 공학적 에이전트 활용 전략 — 5단계

### STEP 1. 대화 전 스펙 문서 작성

**목적:** 에이전트에게 탐색 여지를 주지 않고 명확한 목표를 전달

**실행 방법:**
- `CLAUDE.md` 또는 `PRD.md` 파일에 아래 항목을 사전 정의
  - 기술 스택 (언어, 프레임워크, DB)
  - 구현 대상 기능 목록
  - 완료 기준 (Done Criteria)
  - 제약 조건 (사용하지 않을 라이브러리, 성능 요구 등)

---

### STEP 2. 기능을 원자 단위로 분해

**원칙:** 한 세션 = 한 기능 (기능 욕심 금지)

**올바른 분해 예시:**

```
❌ 잘못된 접근: "로그인, 회원가입, 피드, 프로필, 설정 화면 모두 만들어줘"

✅ 올바른 접근:
  세션 1: 이메일 로그인 (Supabase Auth)
  세션 2: 회원가입 + 이메일 인증
  세션 3: 피드 리스트 (무한 스크롤, 페이지당 20개)
  세션 4: 프로필 조회 + 수정
  세션 5: 설정 화면
```

---

### STEP 3. 세션 분리 전략

**원칙:** 기능 완료 시 반드시 새 대화 시작

```
[기능 완료] → 코드 파일 저장 → 새 대화 시작 → 컨텍스트 브리핑 붙여넣기 → 다음 기능 시작
```

**금지 행동:**
- 한 대화에서 여러 기능을 연속으로 요청
- 에러 해결을 같은 대화에서 10번 이상 반복 (새 세션으로 전환)

---

### STEP 4. 컨텍스트 브리핑 압축

**원칙:** 전체 코드 붙여넣기 금지 → 요약 브리핑만 전달

**세션 시작 템플릿 (매 세션 붙여넣기용):**

```markdown
## 프로젝트 현황

📁 기술 스택: [React Native / Flutter / 기타] + [Supabase / Firebase] + [TypeScript / Dart]
✅ 완료된 기능: [이전 세션에서 완료한 기능 목록]
🔧 지금 작업할 기능: [이번 세션에서 구현할 단 하나의 기능]
📌 제약 조건: [사용 금지 라이브러리, 성능 요건 등]
⚠️ 파일 위치: [신규 생성 또는 수정할 파일 경로]
🎯 완료 기준: [이 기능이 완료되었다고 판단하는 조건]

→ 위 내용을 기반으로 구현 계획을 3줄로 요약해줘. 확인 후 코딩 시작.
```

---

### STEP 5. 검증 체크포인트 설정

**원칙:** 코드 생성 전 계획 먼저 확인, 승인 후 구현

**진행 순서:**

```
1. 브리핑 전달 → "계획 먼저 요약해줘"
2. 계획 확인 → 방향이 맞으면 "진행해줘" / 틀리면 수정 요청
3. 구현 완료 → 동작 확인
4. 완료 후 → 새 세션 시작
```

## 6. MVP 개발 단계별 기능 구현 순서

### 6.1 권장 기술 스택 (초기 MVP)

| 레이어 | 추천 | 이유 |
|--------|------|------|
| 프론트엔드 | React Native 또는 Flutter | 크로스 플랫폼, 에이전트 코드 품질 우수 |
| 백엔드 | Supabase | 인증·DB·Storage 통합, 설정 최소화 |
| 언어 | TypeScript (RN) / Dart (Flutter) | 타입 오류 에이전트가 자동 탐지 가능 |

### 6.2 기능 구현 권장 순서

```
1단계: 인증 (로그인 / 회원가입)
  └── 에이전트 지시: "Supabase Auth 이메일 로그인, 에러 처리 포함"

2단계: 핵심 데이터 모델 + DB 스키마
  └── 에이전트 지시: "users, posts 테이블 스키마 설계 및 Supabase 마이그레이션 파일"

3단계: 핵심 CRUD 기능 1개
  └── 에이전트 지시: "피드 리스트 조회 API 연동, 무한 스크롤, 로딩 스피너"

4단계: 기본 UI 화면 흐름
  └── 에이전트 지시: "로그인→피드→상세 화면 네비게이션 구조"

5단계: 외부 연동 (선택)
  └── 에이전트 지시: "결제 / 푸시 알림 / 이미지 업로드 중 1개"
```

## 8. 토큰 절약 실천 체크리스트

세션 시작 전:
- [ ] 이번 세션에서 구현할 기능이 단 하나로 확정되었는가?
- [ ] 컨텍스트 브리핑 템플릿을 작성했는가?
- [ ] 완료 기준이 명확히 정의되었는가?

세션 진행 중:
- [ ] 에이전트에게 계획 요약을 먼저 요청했는가?
- [ ] 전체 코드 파일을 대화에 붙여넣지 않았는가?
- [ ] 동일 에러를 5회 이상 같은 세션에서 반복하지 않았는가?

세션 종료 후:
- [ ] 완성된 코드를 파일로 저장했는가?
- [ ] 다음 세션을 위한 브리핑 요약을 갱신했는가?
- [ ] 새 대화를 시작했는가?

## 9. 자주 발생하는 실수와 대응

| 실수 | 증상 | 대응 |
|------|------|------|
| 세션 한 곳에서 너무 많은 기능 요청 | 토큰 임박, 응답 품질 저하 | 기능 단위로 새 세션 시작 |
| 전체 코드 파일 붙여넣기 | 즉시 토큰 급증 | 파일 경로와 변경 부분만 전달 |
| 에러 반복 수정 무한 루프 | 같은 에러가 계속 반복됨 | 새 세션에서 에러 메시지만 요약해 전달 |
| 완료 기준 없이 "더 잘 만들어줘" | 방향 없는 반복 수정 | Done Criteria 명시 후 재요청 |
| 아키텍처 결정을 에이전트에게 위임 | 일관성 없는 코드 구조 | 구조는 개발자가 결정, 구현만 위임 |
